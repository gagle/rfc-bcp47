# RFC 5646 — Tags for Identifying Languages

- **Full title**: Tags for Identifying Languages
- **URL**: https://www.rfc-editor.org/rfc/rfc5646
- **Status**: Best Current Practice (BCP 47)
- **Obsoletes**: RFC 4646
- **Updated by**: RFC 6067, RFC 6497

## Scope

Defines the syntax, semantics, and registration of language tags used to identify human languages in Internet protocols. This is the core specification that this library implements.

## Section 2.1 — Syntax (ABNF Grammar)

### Langtag Production

```abnf
Language-Tag  = langtag / privateuse / grandfathered

langtag       = language
                ["-" script]
                ["-" region]
                *("-" variant)
                *("-" extension)
                ["-" privateuse]
```

Ordering is strict — components MUST appear in this sequence.

### Language Subtag

```abnf
language      = 2*3ALPHA ["-" extlang]   ; shortest ISO 639 code
              / 4ALPHA                    ; reserved for future use
              / 5*8ALPHA                  ; registered language subtag
```

- **2-3 chars**: ISO 639-1 (2-letter) or ISO 639-2/3/5 (3-letter)
- **4 chars**: Reserved for possible future standardization — syntactically valid per ABNF but no values currently registered
- **5-8 chars**: IANA-registered language subtags
- **Private use range**: `qaa` through `qtz`

### Extended Language Subtag

```abnf
extlang       = 3ALPHA *2("-" 3ALPHA)    ; max 3 in sequence
```

- Exactly 3 alphabetic characters per subtag
- Max 3 total (1 primary + 2 additional)
- Each has a required `Prefix` field in the IANA registry
- Extlang MUST NOT appear after 4+ character language subtags
- Extlang records MUST NOT include another extlang in their Prefix

### Script Subtag

```abnf
script        = 4ALPHA                   ; ISO 15924 code
```

- Exactly 4 alphabetic characters
- Canonical form: Titlecase (first uppercase, rest lowercase)
- At most ONE per tag
- SHOULD be omitted when it matches the language's `Suppress-Script` field
- Private use range: `Qaaa` through `Qabx`

### Region Subtag

```abnf
region        = 2ALPHA                   ; ISO 3166-1 code
              / 3DIGIT                   ; UN M.49 code
```

- 2-letter codes: ISO 3166-1 alpha-2 (canonical: UPPERCASE)
- 3-digit codes: UN M.49 numeric (no case transformation)
- At most ONE per tag
- Private use ranges: `AA`, `QM`-`QZ`, `XA`-`XZ`, `ZZ`

### Variant Subtag

```abnf
variant       = 5*8alphanum             ; letter-starting
              / (DIGIT 3alphanum)        ; digit-starting (4 chars)
```

- Letter-starting: 5-8 alphanumeric characters
- Digit-starting: exactly 4 characters (1 digit + 3 alphanumeric)
- The same variant MUST NOT appear more than once in a tag
- Multiple variants allowed in sequence
- Position: after language/extlang/script/region, before extensions

### Extension Subtag

```abnf
extension     = singleton 1*("-" (2*8alphanum))

singleton     = DIGIT / %x41-57 / %x59-5A / %x61-77 / %x79-7A
```

- Singleton: single alphanumeric character (0-9, A-Z excluding X)
- `x` is reserved for privateuse — NOT a valid extension singleton
- Extension subtags: 2-8 alphanumeric characters each
- At least one subtag per singleton required
- Multiple extensions per tag allowed (different singletons)
- The same singleton MUST NOT appear more than once

### Privateuse

```abnf
privateuse    = "x" 1*("-" (1*8alphanum))
```

- Reserved singleton: always `x`
- Subtags: 1-8 alphanumeric characters each
- Always the final component in a langtag
- Can also appear as a standalone tag: `x-whatever`

### Grandfathered Tags

```abnf
grandfathered = irregular / regular
```

Complete enumerated lists — not generated by the grammar.

## Section 2.1.1 — Case Sensitivity

Tags are case-insensitive for matching. Canonical case conventions:

| Subtag type | Canonical case | Example |
|-------------|---------------|---------|
| Language | lowercase | `en`, `zh` |
| Extlang | lowercase | `cmn`, `yue` |
| Script | Titlecase | `Latn`, `Hans` |
| Region (2-letter) | UPPERCASE | `US`, `GB` |
| Region (3-digit) | unchanged | `001`, `419` |
| Variant | lowercase | `1996`, `valencia` |
| Extension singleton | lowercase | `u`, `t` |
| Extension subtags | lowercase | `ca`, `buddhist` |
| Privateuse singleton | lowercase | `x` |
| Privateuse subtags | lowercase | `custom` |

3. `i-bnn`
4. `i-default`
5. `i-enochian`
6. `i-hak`

14. `i-tsu`
15. `sgn-BE-FR`
16. `sgn-BE-NL`
17. `sgn-CH-DE`

### Regular (9 tags)

Tags that appear valid but have non-standard semantics:

1. `art-lojban`
2. `cel-gaulish`
3. `no-bok`
4. `no-nyn`
5. `zh-guoyu`

Many have `Preferred-Value` entries pointing to modern equivalents (e.g., `zh-min-nan` → `nan`).

## Section 4.5 — Canonicalization

The RFC defines 3 canonicalization steps:

1. **Extension sequences are ordered** into case-insensitive ASCII order by singleton subtag
2. **Redundant or grandfathered tags** are replaced by their `Preferred-Value`, if there is one
3. **Subtags are replaced** by their `Preferred-Value`, if there is one. For extlangs, the original primary language subtag is also replaced if there is a primary language subtag in the `Preferred-Value`

> **Implementation note:** This library implements a 6-step canonicalization that combines §4.5 (the 3 steps above), §2.1.1 (case conventions — described as optional by the RFC: "processors MAY regularize the case"), §4.1 (suppress-script removal — a SHOULD recommendation), and extension-specific canonicalization from UTS #35 (via RFC 6067 and RFC 6497). The 6 implementation steps are:

### Step 1: Case Normalization (from §2.1.1 — optional per RFC)

Apply the case conventions from §2.1.1 (table above).

### Step 2: Deprecated Subtag Replacement (RFC §4.5 steps 2-3)

Replace any subtag that has a `Preferred-Value` in the IANA registry:
- Deprecated language subtags (e.g., `iw` → `he`)
- Deprecated region subtags (e.g., `BU` → `MM`)
- Deprecated variant subtags (e.g., `heploc` → `alalc97`)
- Grandfathered tags with `Preferred-Value` (e.g., `i-klingon` → `tlh`)
- Redundant tags with `Preferred-Value` (e.g., `sgn-BR` → `bzs`)

### Step 3: Extlang Promotion (RFC §4.5 step 3)

If the tag contains `language-extlang` where the extlang's `Prefix` matches the language, replace both with just the extlang's `Preferred-Value`:
- `zh-cmn` → `cmn`
- `zh-yue` → `yue`

### Step 4: Suppress-Script Removal (from §4.1 — SHOULD per RFC)

If the script matches the language's `Suppress-Script` field, remove it:
- `en-Latn` → `en` (Latn is suppressed for en)
- `ru-Cyrl` → `ru` (Cyrl is suppressed for ru)

> **Note:** Suppress-script removal is not part of §4.5. It is a SHOULD recommendation from §4.1: "script subtag SHOULD NOT be used to form language tags when... the Suppress-Script field is defined."

### Step 5: Extension Ordering (RFC §4.5 step 1)

Sort extension sequences by singleton in case-insensitive ASCII order:
- `en-u-ca-buddhist-a-foo` → `en-a-foo-u-ca-buddhist`

### Step 6: Extension-Specific Canonicalization (from UTS #35)

- **U extension** (UTS #35 / RFC 6067): Sort attributes in US-ASCII order, sort keywords by key
- **T extension** (UTS #35 / RFC 6497): Sort fields by tkey alphabetically

> **Note:** RFC 5646 §4.5 generically states "An extension MUST define any relationships... and thus MAY define an alternate canonicalization scheme." It does not mention U or T extensions by name — those came later in RFC 6067 and RFC 6497, with detailed rules defined in UTS #35.

## Section 4.4 — Variant Subtags

- Uniqueness: same variant MUST NOT repeat within a tag
- Order: registry `Prefix` fields suggest (but don't mandate) ordering
- Mutually exclusive variants should not be combined (e.g., conflicting spelling reforms)
