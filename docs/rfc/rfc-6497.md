# RFC 6497 — BCP 47 Extension T (Transformed Content)

- **Full title**: BCP 47 Extension T
- **URL**: https://www.rfc-editor.org/rfc/rfc6497
- **Status**: Informational
- **Extends**: RFC 5646

## Scope

Defines the `t` extension singleton for BCP 47 language tags, used to indicate transformed content — the source language/script from which content was derived and the mechanism of transformation.

## Syntax

> **Source note:** The grammar below uses UTS #35 rule names (`tlang`, `tkey`, `tvalue`). RFC 6497 uses different names: `lang` (not `tlang`) and `fsep` (not `tkey`). The structural grammar is equivalent. See `docs/rfc/uts-35.md` for the complete EBNF.

```ebnf
transformed_extensions
    = sep [tT]
      ((sep tlang (sep tfield)*)
      | (sep tfield)+) ;

tlang   = unicode_language_subtag
          (sep unicode_script_subtag)?
          (sep unicode_region_subtag)?
          (sep unicode_variant_subtag)* ;

tfield  = tkey tvalue ;
tkey    = alpha digit ;               ; first alphabetic, second digit (RFC 6497: "fsep")
tvalue  = alphanum{3,8} (sep alphanum{3,8})+ ;
```

> **Subtag count discrepancy:** UTS #35 defines `tvalue` with `+` (2+ subtags minimum). RFC 6497's ABNF uses `1*("-" 3*8alphanum)` (1+ subtags minimum). This library follows RFC 6497's 1+ minimum. See `docs/rfc/uts-35.md` for details.

### Source Language (tlang)

- Optional — may be omitted if fields don't require it
- Must be a valid, canonical BCP 47 language tag (not a full tag — no extensions or privateuse)
- Structure: `language [-script] [-region] *[-variant]`
- Irregular productions prohibited
- Use `und` when only script matters (e.g., `t-und-Cyrl`)

### Fields

- **tkey** (RFC 6497: `fsep`): exactly 2 characters — first alphabetic, second digit (e.g., `m0`, `d0`)
- **tvalue**: 3-8 alphanumeric subtags following the tkey
- Identical tkeys cannot repeat within a single tag

### Tkey Pattern

The tkey regex: `[a-z]\d` — first character is alphabetic, second is a digit. This constraint (from UTS #35: `tkey = alpha digit`) ensures disjointness with U extension ukeys (`alphanum alpha`).

All 8 tkeys defined in UTS #35 (only `m0` appears in RFC 6497 itself; the others come from CLDR — see `docs/rfc/uts-35.md` for complete type value tables):
- `d0` — destination (output transformation, e.g., `fwidth`, `ascii`)
- `h0` — hybrid locale request
- `i0` — input method engine (e.g., `pinyin`, `handwrit`)
- `k0` — keyboard transform (e.g., `qwerty`, `dvorak`)
- `m0` — mechanism/transform engine (e.g., `true`, `ungegn`) — *the only tkey in RFC 6497*
- `s0` — source (input transformation, e.g., `ascii`, `npinyin`)
- `t0` — machine translation
- `x0` — private use transform

## Section 2.3 — Canonical Form

1. All subtags lowercase
2. **Fields sorted alphabetically by tkey** (separator)
3. Order of subtags within a field is significant — MUST NOT be changed
4. Source language (tlang) appears first, before any fields

### Canonicalization Example

Input: `en-t-ja-m0-true-d0-fwidth`
Output: `en-t-ja-d0-fwidth-m0-true`

- Source `ja` preserved first
- Fields `m0-true`, `d0-fwidth` → sorted by tkey to `d0-fwidth`, `m0-true`

## Validation

- Field separators (tkeys) must not repeat
- Source language must be a valid canonical langtag (without extensions)
- Cannot contain single-character subtags (would start a new extension)
