# RFC 6067 — BCP 47 Extension U (Unicode Locale)

- **Full title**: BCP 47 Extension U
- **URL**: https://www.rfc-editor.org/rfc/rfc6067
- **Status**: Informational
- **Extends**: RFC 5646

## Scope

Defines the `u` extension singleton for BCP 47 language tags, used to convey Unicode locale information (calendar, collation, numbering system, etc.). Derived from CLDR (Unicode Common Locale Data Repository).

## Syntax

> **Source note:** RFC 6067 describes the U extension structure in prose only and does not contain formal ABNF. The grammar below is from UTS #35 (Unicode LDML), which is the authoritative source for U extension syntax. See `docs/rfc/uts-35.md` for the complete EBNF.

```ebnf
unicode_locale_extensions
    = sep [uU]
      ((sep keyword)+
      |(sep uattribute)+ (sep ufield)*) ;

ufield    = ukey (sep uvalue)? ;
ukey      = alphanum alpha ;          ; first alphanumeric, second alphabetic
uvalue    = alphanum{3,8} (sep alphanum{3,8})* ;
uattribute = alphanum{3,8} ;
```

Note: `keyword` is an alias for `ufield`, `key` for `ukey`, `type` for `uvalue`, `attribute` for `uattribute`.

### Attributes

- 3-8 alphanumeric characters
- Appear first, before any keywords
- Example: `en-u-attr1-attr2-ca-buddhist`

### Keywords

- **Key (ukey)**: exactly 2 characters (first char: alphanumeric, second char: alphabetic)
- **Type (uvalue)**: optional, 3-8 alphanumeric characters per subtag (can have multiple subtags)
- A key MUST NOT appear more than once
- Only the first occurrence of a duplicate key conveys meaning
- When a key has no type value, the value `"true"` is implied (UTS #35)

### Key Pattern

The key regex: `[a-z\d][a-z]` — first character is alphanumeric, second is alphabetic. This constraint comes from UTS #35 (`ukey = alphanum alpha`), which is narrower than what RFC 6067 describes (ensuring disjointness with T extension tkeys).

Common keys (5 of 29 defined in UTS #35 — see `docs/rfc/uts-35.md` for the complete list):
- `ca` — calendar (e.g., `buddhist`, `gregorian`)
- `co` — collation (e.g., `phonebk`, `standard`)
- `nu` — numbering system (e.g., `latn`, `arab`)
- `cf` — currency format
- `hc` — hour cycle

## Canonical Form

1. All subtags lowercase
2. **Attributes sorted** in US-ASCII order (numbers before letters)
3. **Keywords sorted by key** in US-ASCII order
4. Order of subtags within a keyword type is significant — do NOT rearrange

### Canonicalization Example

Input: `en-u-zebra-apple-co-phonebk-ca-buddhist`
Output: `en-u-apple-zebra-ca-buddhist-co-phonebk`

- Attributes `zebra`, `apple` → sorted to `apple`, `zebra`
- Keywords `co-phonebk`, `ca-buddhist` → sorted by key to `ca-buddhist`, `co-phonebk`

## Validation

- Duplicate attributes: only first occurrence meaningful
- Duplicate keys: only first occurrence meaningful
- All subtag data derived from CLDR data files
